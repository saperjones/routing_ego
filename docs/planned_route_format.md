# Format Spec: `planned_route.json`

The **planned routing path** for an AVP clip: a road-snapped, densified route
that the vehicle is meant to follow, plus the waypoints it was built from and the
source (measured) ego track it was derived from. Produced by the route generator
`gen_route_from_ego.py` (see `route_generation_result/generation.log`).

Location: `route_generation_result/planned_route.json`.

> **This is NOT the measured ego trajectory.** The measured pose/attitude per
> frame lives in `../ego_route_llh.json`. This file is the *planned route* (and
> the track it was generated from). Do not confuse the two.

## ŌÜĀ’ĖÅ DATUM & COORDINATE LAYOUT ŌĆö READ FIRST

- **All coordinates in this file are WGS-84.** (`planned_route`, `ego_track`,
  and `waypoints`.)
- **Every coordinate pair is `[latitude, longitude]` ŌĆö latitude first**,
  degrees. This is the *opposite* order from `ego_route_llh.json`, which uses
  explicit `longitude`/`latitude` fields. Do not swap them.
- The sibling file **`ego_route_llh.json` is GCJ-02**, not WGS-84. This file's
  `ego_track` is that same drive **after** the generator converted it GCJ-02ŌåÆ
  WGS-84 (`generation.log`: *"5047 ÕÄ¤Õ¦ŗńé╣ ŌåÆ 4912 µ£ēµĢłńé╣(ÕĘ▓ĶĮ¼ WGS84)"*). So the two

## 2. Coordinate layout and datum (detail)

- **Datum:** WGS-84 throughout.
- **Order:** `[latitude, longitude]`, degrees, latitude first ŌĆö for every pair in
  `ego_track`, `waypoints`, and `planned_route`.
- **No height / no per-point attitude** in this file ŌĆö it is a 2D geodetic path.
  (Headings are only given as `start_yaw_deg` and per-waypoint `waypoint_yaws`,
  in degrees.)
- Operating area (observed): ~lat 31.8347ŌĆō31.8363, lon 117.1361ŌĆō117.1391
  (Hefei).

## 4. Provenance and pipeline (from `generation.log`)

`planned_route.json` is generated by
`gen_route_from_ego.py <ego_route_llh.json> <planned_route.json>`:

1. Load `ego_route_llh.json`: **5047 raw points ŌåÆ 4912 valid points, converted
   to WGS-84** ŌåÆ becomes `ego_track`.
2. DouglasŌĆōPeucker simplify (`dp_epsilon_m = 6 m`) ŌåÆ a skeleton (~14 points).
3. Extract via-points (leg midpoints + densify; `yaw_turn_threshold_deg`,
   `min_leg_for_waypoint_m`, `min_waypoint_gap_m`) ŌåÆ the 11 `waypoints` with
   `waypoint_yaws`.
4. Estimate the start heading over `start_heading_span_m` ŌåÆ `start_yaw_deg`.
5. Call the `nvai` route planner over a road graph (`roads.json`), constrained
   through the waypoints+headings, snapping within `on_road_threshold_m`, output
   densified at `densify_step_m` ŌåÆ `planned_route` (2067 points).
6. Report lengths: `ego_length_m = 1149.2`, `planned_length_m = 1032.1`.

Because it is snapped to the road network, `planned_route` follows road
centerlines and is shorter than the raw `ego_track`; the two are close but not
identical.

## 5. Relationship to `ego_route_llh.json`

| Aspect | `planned_route.json` (this file) | `ego_route_llh.json` |
|--------|----------------------------------|----------------------|
| Role | Planned route + waypoints + source track | Measured per-frame ego pose/attitude/velocity |
| Datum | **WGS-84** | **GCJ-02** (metadata mislabels it "WGS84") |
| Coordinate order | `[lat, lon]` pairs (latitude first) | explicit `longitude` / `latitude` fields |
| Heading units | degrees (`start_yaw_deg`, `waypoint_yaws`) | radians (`yaw_boot`) |
| Extras | `params`, lengths | velocities, yaw-rate, quaternion, timestamps, boot-frame `x,y,z` |
| Frame rate | not time-sampled (a path) | ~20 Hz, timestamped |

`ego_track` here is derived from `ego_route_llh.json`'s track (GCJ-02ŌåÆWGS-84,
5047ŌåÆ4912). To relate a per-frame ego pose to this route, convert datums first.

## 6. Observed invariants

- Coordinate pairs are `[lat, lon]` (latitude in ~31.83, longitude in ~117.13).
- `planned_route` spacing Ōēł 0.5 m (uniform, = `densify_step_m`).
- `ego_track` spacing Ōēł 0.23 m mean (raw drive, non-uniform).
- `len(waypoints) == len(waypoint_yaws)` (11 each).
- `planned_length_m < ego_length_m` (road-snapped route is shorter).
